# 蓝牙连接使用指南

## 问题修复说明

### 1. ✅ 已修复：Demo模式控制按钮崩溃
**问题**：在demo模式下点击控制按钮导致应用崩溃  
**原因**：Demo模式下outputStream为null，调用write()方法导致NullPointerException  
**修复**：在sendCommand方法中添加demo模式检测，直接返回成功

### 2. ✅ 已修复：蓝牙设备扫描功能
**问题**：无法搜索到蓝牙设备  
**原因**：原实现只扫描已配对设备，没有发现新设备功能  
**修复**：
- 添加蓝牙设备发现功能
- 注册广播接收器监听设备发现事件
- 同时显示已配对和新发现的设备
- 添加完整的权限检查

## 蓝牙连接使用步骤

### 步骤1: 准备STM32蓝牙模块

#### 硬件连接
```
STM32          蓝牙模块(HC-05/HC-06)
PA2 (TX)  -->  RX
PA3 (RX)  <--  TX
VCC       -->  VCC (3.3V或5V)
GND       -->  GND
```

#### 蓝牙模块配置
使用AT命令配置蓝牙模块（通过串口助手）：

```
AT+NAME=SmartCabinet    // 设置设备名称
AT+PSWD=1234            // 设置配对密码
AT+UART=115200,0,0      // 设置波特率115200
AT+ROLE=0               // 设置为从机模式
```

### 步骤2: Android应用权限设置

#### 首次使用需要授予权限：
1. 打开应用
2. 系统会提示请求以下权限：
   - **蓝牙权限**（必需）
   - **位置权限**（必需，Android要求）
   - **通知权限**（可选，用于报警）
3. 点击"允许"授予所有权限

#### 如果权限被拒绝：
1. 打开手机"设置"
2. 找到"应用管理" → "MySmart"
3. 点击"权限"
4. 启用"蓝牙"和"位置"权限

### 步骤3: 搜索蓝牙设备

#### 方法1: 使用应用内扫描
1. 打开MySmart应用
2. 点击底部导航栏的"设备"图标
3. 点击"扫描设备"按钮
4. 等待扫描完成（约12秒）
5. 在列表中找到你的设备（如"SmartCabinet"或"HC-05"）

#### 方法2: 先在系统设置中配对
1. 打开手机"设置" → "蓝牙"
2. 确保蓝牙已开启
3. 在"可用设备"列表中找到你的设备
4. 点击配对（输入密码，默认1234或0000）
5. 配对成功后，打开MySmart应用
6. 点击"扫描设备"，已配对的设备会立即显示

### 步骤4: 连接设备

1. 在设备列表中点击你的设备
2. 点击"连接"按钮
3. 等待连接建立（约3-5秒）
4. 连接成功后，状态显示"已连接"
5. 返回"监控"页面查看实时数据

### 步骤5: 控制设备

连接成功后，在"监控"页面可以：
- 切换风扇开关
- 切换紫外线灯开关
- 切换照明灯开关
- 查看实时传感器数据

## 常见问题排查

### Q1: 扫描不到设备

**检查项**：
1. ✅ 蓝牙是否已开启
   ```
   设置 → 蓝牙 → 确保开关打开
   ```

2. ✅ 权限是否已授予
   ```
   设置 → 应用管理 → MySmart → 权限
   确保"蓝牙"和"位置"权限已启用
   ```

3. ✅ 蓝牙模块是否通电
   ```
   检查蓝牙模块LED是否闪烁
   HC-05: 快速闪烁表示未配对，慢速闪烁表示已配对
   ```

4. ✅ 蓝牙模块是否可被发现
   ```
   在手机系统蓝牙设置中查看是否能看到设备
   如果看不到，可能是模块故障或配置问题
   ```

5. ✅ 距离是否太远
   ```
   蓝牙有效距离约10米
   尽量靠近设备进行扫描
   ```

### Q2: 连接失败

**可能原因及解决方法**：

1. **设备已被其他应用占用**
   ```
   关闭其他可能使用蓝牙的应用
   重启手机蓝牙
   ```

2. **配对密码错误**
   ```
   在系统设置中删除配对
   重新配对，输入正确密码（默认1234或0000）
   ```

3. **蓝牙模块配置错误**
   ```
   检查波特率是否为115200
   检查是否设置为从机模式(ROLE=0)
   ```

4. **STM32未运行或串口未初始化**
   ```
   检查STM32是否正常运行
   检查USART2是否正确初始化
   ```

### Q3: 连接后无数据

**检查项**：

1. ✅ STM32是否正在发送数据
   ```
   使用串口助手连接蓝牙模块
   查看是否收到JSON格式的数据
   ```

2. ✅ 数据格式是否正确
   ```
   数据应为JSON格式：
   {"temp":25.5,"humi":60.3,"tvoc":150,"co2":450,"fan":1,"uv":0,"light":1,"alarm":0}
   ```

3. ✅ 波特率是否匹配
   ```
   STM32: 115200
   蓝牙模块: 115200
   确保两者一致
   ```

### Q4: 控制命令无响应

**检查项**：

1. ✅ STM32是否集成了命令接收功能
   ```
   确认已添加 app_wifi_command.c
   确认main.c中调用了 APP_WiFi_ReceiveCommand()
   ```

2. ✅ 蓝牙模块是否支持双向通信
   ```
   HC-05: 支持
   HC-06: 支持
   其他模块: 需确认
   ```

3. ✅ 连接是否稳定
   ```
   查看连接状态是否为"已连接"
   如果频繁断开，检查电源和距离
   ```

## 蓝牙模块推荐

### HC-05（推荐）
- **优点**：
  - 支持主从模式切换
  - 配置灵活
  - 稳定性好
  - 价格便宜（10-20元）
- **缺点**：
  - 需要AT命令配置
  - 体积稍大

### HC-06
- **优点**：
  - 配置简单
  - 价格便宜（8-15元）
  - 体积小
- **缺点**：
  - 只能作为从机
  - 功能相对简单

### JDY-31（SPP-C）
- **优点**：
  - 支持BLE和经典蓝牙
  - 功耗低
  - 稳定性好
- **缺点**：
  - 价格稍贵（15-25元）
  - 配置复杂

## 蓝牙vs WiFi对比

| 特性 | 蓝牙 | WiFi |
|------|------|------|
| 连接距离 | 10米 | 50-100米 |
| 功耗 | 低 | 高 |
| 配置难度 | 简单 | 复杂 |
| 稳定性 | 好 | 很好 |
| 成本 | 低（10-20元） | 中（20-40元） |
| 推荐场景 | 近距离、移动使用 | 远距离、固定位置 |

## 代码示例

### STM32端（使用蓝牙）

```c
// 在main.c中，USART2连接到蓝牙模块
BSP_USART2_Init(115200);  // 初始化串口

// 主循环中发送数据
while (1) {
    APP_DataCollect(&g_system_state);
    APP_InputScan(&g_system_state);
    APP_WiFi_ReceiveCommand();  // 接收蓝牙命令
    APP_ControlLogic_Process(&g_system_state);
    APP_Display_Update(&g_system_state);
    
    // 通过蓝牙发送数据（使用USART2）
    static uint32_t send_timer = 0;
    send_timer += SYSTEM_LOOP_DELAY_MS;
    if (send_timer >= 5000u) {
        send_timer = 0;
        APP_WiFi_SendData(&g_system_state);  // 发送JSON数据
    }
    
    delay_ms(SYSTEM_LOOP_DELAY_MS);
}
```

### Android端（蓝牙连接）

```java
// 扫描蓝牙设备
viewModel.scanDevices();

// 连接设备
DeviceInfo device = new DeviceInfo("98:D3:31:F5:8C:7A", "SmartCabinet", "BLUETOOTH");
viewModel.connectDevice(device);

// 发送控制命令
viewModel.toggleFanModule();  // 控制风扇
```

## 总结

现在你的应用已经支持：
1. ✅ Demo模式控制（不会崩溃）
2. ✅ 蓝牙设备扫描（已配对+新设备）
3. ✅ 蓝牙设备连接
4. ✅ 蓝牙数据接收（JSON格式）
5. ✅ 蓝牙控制命令发送
6. ✅ WiFi连接（作为备选方案）

选择蓝牙还是WiFi取决于你的使用场景：
- **蓝牙**：适合近距离、移动使用、低功耗场景
- **WiFi**：适合远距离、固定位置、需要网络功能的场景

两种方式都使用相同的JSON协议，可以无缝切换！
